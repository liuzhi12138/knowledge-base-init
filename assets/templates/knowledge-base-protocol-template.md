# {项目名} 知识库维护协议

> 本文件定义 {项目名} 项目 AI Coding 知识库的增量维护规则。
> 所有 AI 在更新知识库前 **必须阅读本文件**。

---

## 一、增量更新触发

完成非简单对话、源码扫描或业务代码修改后，执行一次**知识库增量判断**。

仅当发现可复用的新信息时，向用户展示候选项并询问是否增量归纳到知识库。候选项包括：

| 类型 | 默认落点 |
|------|--------|
| 已确认业务规则 | `docs_kb_{项目名}/context/04-business-rules/{模块}-rules.md` |
| 源码事实 | `docs_kb_{项目名}/context/04-business-rules/{模块}/source-facts.md` |
| 历史旁路 | `docs_kb_{项目名}/context/04-business-rules/{模块}/legacy-and-side-paths.md` |
| 治理缺口 | `docs_kb_{项目名}/context/04-business-rules/{模块}/governance-gaps.md` |
| 术语与同义词 | `docs_kb_{项目名}/context/GLOSSARY.md` |

`AGENTS_KB_{项目名}.md`、`00-index.md`、Checklist 和模板属于控制面文件，候选修改需单独列出。

**未经用户明确确认，不得修改知识库。** 用户拒绝或跳过时，不落盘、不创建草稿。源码扫描只能更新源码事实和治理缺口，不能自动升级为团队业务规则。

普通问答、翻译、简单命令执行以及未产生新结论的代码阅读，不询问知识库归纳。

候选提示格式：

```text
本轮发现 N 项可沉淀内容：
1. 源码事实：...
2. 治理缺口：...
3. 术语同义词：...

是否将这些候选项增量归纳到知识库？
```

## 二、分支安全约束

{如果项目使用 Git 分支管理，填写以下内容；否则删除本节}

**基准分支**：`{基准分支名}`（基准 commit：`{commit-hash}`，{日期}）。知识库文档反映的是基准分支上的代码状态。

**文档分级与更新权限**：

| 文档类型 | 分支敏感度 | 基准分支上的权限 | 非基准分支上的权限 |
|---------|-----------|-----------------|------------------|
| `{模块}-rules.md` | 低 — 业务规则稳定 | 经用户确认后可更新 | **只读**，不得覆写 |
| `legacy-and-side-paths.md` | 低 — 历史不会消失 | 经用户确认后可更新 | **只读**，不得覆写 |
| `governance-gaps.md` | 中 — 新分支可能修复缺口 | 可全量刷新 | **只追加**新缺口，不删除已有条目 |
| `source-facts.md` | **高** — 代码差异大 | 可全量刷新（需更新元数据头） | **只读**，不得覆写 |
| `change-checklist.md` | 低 — 验收标准稳定 | 经用户确认后可更新 | **只读**，不得覆写 |
| 基础设施文档（01~07） | 中 | 经用户确认后可更新 | **只读**，不得覆写 |
| `GLOSSARY.md` | 低 | 可增量追加术语 | 可增量追加术语，不删除已有条目 |
| `00-index.md` / `AGENTS_KB_{项目名}.md` | 低 | 经用户确认后可更新 | **只读**，不得覆写 |

**操作前必检流程**：

```
1. 执行 `git branch --show-current` 获取当前分支
2. 如果当前分支 = {基准分支} → 按正常流程操作
3. 如果当前分支 ≠ {基准分支} →
   a. 仅允许：追加 governance-gaps 条目、追加 GLOSSARY 术语
   b. 禁止：覆写 rules.md / source-facts.md / legacy / checklist / 基础设施文档
   c. 如需全量刷新知识库，提示用户切换到基准分支
```

## 三、元数据标记

每个 `{模块}-rules.md` 和 `{模块}/source-facts.md` 文件顶部包含元数据注释：

```html
<!-- base-branch: {基准分支} | base-commit: {commit} | last-verified: {日期} -->
```

- AI 在基准分支上全量刷新文档时，必须同时更新 `base-commit` 和 `last-verified`
- AI 在非基准分支上读到元数据头时，可判断文档的基准来源，避免盲目覆写

## 四、显式唤起

需要主动维护知识库时，可以直接使用以下提示词：

```text
扫描 {模块名} 模块，并执行一次知识库 PDCA。
```

```text
检查本轮对话是否有可沉淀内容，执行知识库增量判断。
```

```text
将刚才确认的业务规则归纳到知识库。
```

```text
扫描项目源码，检查知识库是否存在漂移，只生成候选项，不要直接修改。
```

推荐在开发任务结束时补一句 `完成验证后，执行知识库增量判断`；模块治理时使用 `扫描 [模块名]，对照现有知识库执行 PDCA，只列出新增事实、治理缺口和术语候选项`。
